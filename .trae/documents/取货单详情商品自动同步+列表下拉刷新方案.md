# 取货单详情商品自动同步 + 列表下拉刷新方案

## 一、问题分析

### Bug：切换 PDA 后取货单详情商品不显示

**现象**：同一账号在 PDA2 登录后，PDA1 创建的取货单详情中商品无法显示。

**根因**：
- `PickDetailViewModel.init`（[PickDetailViewModel.kt](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L100-L103)）仅调用 `loadOrder()`（读 Room 本地数据库）和 `loadSuppliers()`（从 API 获取供应商列表）
- `items` 数据流（[L60-L62](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L60-L62)）由 Room 的 `Flow` 驱动，仅返回本地数据库中的明细
- 另一个 PDA 添加的明细存在于**后端数据库**中，但从未同步到当前 PDA 的本地 Room 数据库
- 后端 `get_order` API（[orders.py#L146-L183](file:///d:/trea项目/快麦取货通/backend/app/routers/orders.py#L146-L183)）的明细查询**无用户过滤**（`SELECT * FROM pick_items WHERE order_id = ?`），后端数据完整
- `syncItemsFromBackend()`（[L396-L432](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L396-L432)）**仅同步 items，不同步 order 信息** — 其他人领取公共订单时，订单信息（单号、进度条）不会显示

**后端访问控制已验证**（`_check_order_access` [orders.py#L544-L556](file:///d:/trea项目/快麦取货通/backend/app/routers/orders.py#L544-L556)）：
- `assigned_to == username OR created_by == username` → 同一账号在 PDA2 登录时 username 相同，访问通过
- 发布到公共 + 未领取 → 也通过

### 增强：取货单列表缺少下拉刷新

**现象**：其他 PDA 发布到公共的取货单，当前 PDA 无法及时获取。

**根因**：
- `PickListScreen` 的进行中列表（[L161-L224](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/picklist/PickListScreen.kt#L161-L224)）无 PullToRefreshBox
- `PickListViewModel.loadActiveOrders()`（[L80-L92](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/picklist/PickListViewModel.kt#L80-L92)）未管理 `_isLoading` 状态
- 已有的 `ON_RESUME` 生命周期监听（[L112-L122](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/picklist/PickListScreen.kt#L112-L122)）仅触发 `refreshActiveOrders()`，但无加载指示器

## 二、修改内容

### 修改 1：PickDetailViewModel.kt — init 块添加自动同步

**文件**：`app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt`

**改动 A**：在 init 块末尾添加 `syncItemsFromBackend()` 调用

```kotlin
init {
    loadOrder()
    loadSuppliers()
    viewModelScope.launch {  // ← 新增
        syncItemsFromBackend()
    }
}
```

**改动 B**：在 `syncItemsFromBackend()` 中同步 order 信息，确保其他人领取公共订单时订单信息也能正常显示

```kotlin
private suspend fun syncItemsFromBackend() {
    try {
        val token = userRepository.getToken()
        val detail = orderApiService.getOrderDetail(token, orderId)

        // 新增：同步order信息（_order是MutableStateFlow，不是Room Flow，
        // 必须同时更新Room和StateFlow，确保UI能正确反映订单信息）
        val orderEntity = PickOrderEntity(
            id = detail.id,
            orderNo = detail.orderNo,
            status = detail.status,
            completionType = detail.completionType,
            totalCount = detail.totalCount,
            completedCount = detail.completedCount,
            createdAt = TimeUtils.parseBeijingTime(detail.createdAt).let { if (it > 0) it else TimeUtils.now() },
            completedAt = TimeUtils.parseBeijingTimeOrNull(detail.completedAt),
            expireAt = TimeUtils.parseBeijingTime(detail.expireAt).let { if (it > 0) it else TimeUtils.now() + TimeUtils.DEFAULT_EXPIRE_MS },
            createdBy = detail.createdBy,
            assignedTo = detail.assignedTo,
            visibility = detail.visibility
        )
        pickOrderRepository.insertOrder(orderEntity)  // ← insertOrder (ON CONFLICT REPLACE) 兼容不存在和已存在的场景
        _order.value = orderEntity  // ← 直接设置StateFlow，让UI立即响应

        // 同步明细数据（items由Room Flow自动驱动，无需手动设置）
        detail.items.forEach { itemResponse ->
            // ... 不变
        }
        loadSuppliers()
    } catch (e: Exception) {
        Log.w(TAG, "syncItemsFromBackend失败: ${e.message}")
    }
}
```

**为什么**：
- `_order`（[L56-L57](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L56-L57)）是 `MutableStateFlow`，不是 Room Flow 驱动 — `loadOrder()` 是一次性查询后手动设置 `_order.value`
- 即使 `insertOrder()` 保存到 Room，`_order.value` 仍保持 null（另一个协程的加载结果已在之前写入 null）
- 必须在 `syncItemsFromBackend()` 中直接设置 `_order.value = orderEntity`，确保 UI 能立即响应
- **`insertOrder()` 优于 `updateOrder()`**：`PickOrderDao.insert()`（[L45-L46](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/data/db/dao/PickOrderDao.kt#L45-L46)）使用 `@Insert(onConflict = OnConflictStrategy.REPLACE)`，新设备上订单不在 Room 时也能成功保存；`@Update` 仅更新已存在行，新设备上静默失败。
- `items` 数据流（[L60-L62](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L60-L62)）是由 Room 的 `Flow` + `stateIn` 驱动的，Room 写入后会自动刷新 UI，不需要手动设置
- 与 `refresh()`（[L299-L314](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L299-L314)）的 order 同步逻辑一致，复用 `PickOrderEntity` 构造模式
- 已有 try-catch 保护（[L429-L431](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailViewModel.kt#L429-L431)），网络失败仅 Log.w，不影响页面加载

### 修改 2：PickListViewModel.kt — loadActiveOrders() 增加加载状态

**文件**：`app/src/main/java/com/kuaimai/pda/ui/picklist/PickListViewModel.kt`

**改动**：在 `loadActiveOrders()` 中管理 `_isLoading` 状态

```kotlin
fun loadActiveOrders() {
    viewModelScope.launch {
        _isLoading.value = true  // ← 新增
        try {
            val token = userRepository.getToken()
            val response = orderApiService.listOrders(token, status = 0)
            _activeOrders.value = response.data.map { it.toOrderEntity() }
        } catch (e: Exception) {
            Log.e("PickListViewModel", "加载取货单列表失败: ${e.message}", e)
            _errorMessage.value = "加载取货单列表失败: ${e.message?.take(80) ?: "未知错误"}"
            _activeOrders.value = emptyList()
        } finally {
            _isLoading.value = false  // ← 新增
        }
    }
}
```

**为什么**：
- `_isLoading` 已在 ViewModel 中声明（[L56-L57](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/picklist/PickListViewModel.kt#L56-L57)），但 `loadActiveOrders()` 未设置它
- 其他方法（`createOrder`、`publishOrder`、`claimOrder`）已使用 `_isLoading`，此为补齐
- `_isLoading` 将驱动 PullToRefreshBox 的刷新指示器

### 修改 3：PickListScreen.kt — 添加 PullToRefreshBox

**文件**：`app/src/main/java/com/kuaimai/pda/ui/picklist/PickListScreen.kt`

**改动 A**：新增 import

```kotlin
import androidx.compose.material3.pulltorefresh.PullToRefreshBox
```

**改动 B**：用 PullToRefreshBox 包裹进行中列表内容

```kotlin
// 修改前 — 进行中列表
Column(
    modifier = Modifier
        .fillMaxSize()
        .padding(innerPadding)
) {
    if (activeOrders.isEmpty()) { ... }
    else { ... LazyColumn ... }

// 修改后
PullToRefreshBox(
    isRefreshing = isLoading,
    onRefresh = { viewModel.refreshActiveOrders() },
    modifier = Modifier
        .fillMaxSize()
        .padding(innerPadding)
) {
    Column(
        modifier = Modifier
            .fillMaxSize()
    ) {
        if (activeOrders.isEmpty()) { ... }
        else { ... LazyColumn ... }
    }
}
```

**为什么**：
- 遵循 [PickDetailScreen.kt#L204-L212](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/pickdetail/PickDetailScreen.kt#L204-L212) 的现有 PullToRefreshBox 模式
- `isRefreshing = isLoading` — 使用 ViewModel 的加载状态驱动指示器
- `onRefresh = { viewModel.refreshActiveOrders() }` — `refreshActiveOrders()`（[PickListViewModel.kt#L95](file:///d:/trea项目/快麦取货通/app/src/main/java/com/kuaimai/pda/ui/picklist/PickListViewModel.kt#L95)）直调 `loadActiveOrders()`，此时 `_isLoading` 已管理
- `innerPadding` 从 Column 上移到 PullToRefreshBox，内部 Column 仅 `fillMaxSize()` — 与 PickDetailScreen 一致

## 三、回归风险分析

### PullToRefreshBox 兼容

| 检查项 | 状态 |
|--------|:----:|
| `PullToRefreshBox` 是否可用 | Material3 1.3+，项目已在 PickDetailScreen 使用 |
| `isRefreshing` 不与其他操作冲突 | `_isLoading` 被 `createOrder`/`publishOrder`/`claimOrder` 共享，但均为串行操作+finally 释放 |
| 已完成列表不受影响 | PullToRefreshBox 仅包裹进行中列表分支 |
| 空状态显示正常 | 空状态 Box 在 PullToRefreshBox 内部，`Modifier.fillMaxSize()` 确保居中 |

### 无回归清单

- [x] 不涉及后端代码修改（后端 API 已经正确，无需变更）
- [x] 不涉及数据库 schema 变更
- [x] 不涉及 Room entity/dao 变更
- [x] 不涉及导航/路由变更
- [x] 不涉及其他 ViewModel 或 Screen
- [x] `syncItemsFromBackend()` 新增 order 同步：使用 `insertOrder()`（REPLACE 语义）而非 `updateOrder()`（仅更新）— 新设备也能成功保存
- [x] `insertOrder()` 是 upsert 语义，不影响已存在的 order（主键冲突时 REPLACE）
- [x] PullToRefreshBox 遵循项目已有模式

### 数据流安全（更新）

| 环节 | 验证 | 风险等级 |
|------|------|:--------:|
| `syncItemsFromBackend()` 异常 | try-catch 保护，失败仅 Log.w | 低 |
| 重复 SKU 处理 | Room 有 `UNIQUE(order_id, sku_outer_id)` 约束 + `ON CONFLICT REPLACE` | 低 |
| `items` Flow 自动刷新 | Room 的 `stateIn` 驱动 UI，无手动管理 | 低 |
| 供应商列表刷新 | `syncItemsFromBackend()` 末尾调用 `loadSuppliers()` | 低 |
| init 并发时序 | `loadOrder()` 读 order 表，`syncItemsFromBackend()` 读写 orders+items 表，无竞态 | 低 |
| **`insertOrder()` vs `updateOrder()`** | `PickOrderDao.insert()` 使用 `@Insert(onConflict = REPLACE)`，新设备/新用户也能正常写入✅ | 原 `@Update` 会静默失败❗现已修复 |
