### 变更3：取货单详情供应商筛选按供应商序号排序

**前置条件检查** ⚠️
- ✅ 后端 `get_suppliers` 返回纯名称列表，未排序
- ✅ `PickItemEntity.supplierCode` 存在且为顺序数字格式（如 "0001"）
- ❗ `SELECT DISTINCT supplier_name ... ORDER BY supplier_code` 在 SQLite 中行为不确定（DISTINCT 后 ORDER BY 非选择列） — **需要改用量化写法**

**变更内容**：

| 文件 | 操作 |
|:----|:----|
| [orders.py](file:///d:/trea项目/快麦取货通/backend/app/routers/orders.py) | `get_suppliers` SQL 改为 `GROUP BY supplier_name ORDER BY MIN(supplier_code) ASC` |

SQL 修改前后对比：
```sql
-- 修改前（无序）
SELECT DISTINCT supplier_name FROM pick_items WHERE order_id = ? AND supplier_name != ''
-- 修改后（按编码排序）
SELECT supplier_name FROM pick_items WHERE order_id = ? AND supplier_name != '' GROUP BY supplier_name ORDER BY MIN(supplier_code) ASC
```

## 三、假设与决策

| 决策 | 说明 |
|:----|:------|
| CameraScanScreen 直接删除 | CameraX/MLKit 所有导入均在该文件内，无其他消费者 |
| proguard CameraX 规则同步删除 | 删除 CameraScanScreen 后无需再 keep CameraX 类 |
| SetupQrParser 同步删除 | 仅被 GuideScreen 引用用于二维码解析配置，无其他用途 |
| 扫码方式值从0-2改为0-1 | 旧配置 `scanMethod=1`（相机）升级后变为"手动输入"，合理降级 |
| 供应商排序在后端实现 | 避免 Android 端重复写排序逻辑，使用 `GROUP BY + MIN(code)` 确保确定性 |
| ProductScreen.kt 中 ConfirmDialog 保留 | 供应商切换仍需确认弹窗，仅备注修改去掉确认步骤 |
| encryptedPrefs 从 AppNavigation 移除 | 仅用于 GuideScreen，无其他用途；同时清理 MainActivity 中对应字段和 import |
