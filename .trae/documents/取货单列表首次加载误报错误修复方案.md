# 取货单列表首次加载误报错误修复方案

## 一、根因分析

### 竞态条件 (Race Condition)

取货单列表页面进入时会**同时发起两次**取货单列表请求：

```
时序图:

[ViewModel init]           [DisposableEffect]
        │                         │
 loadActiveOrders()         注册 ON_RESUME 观察者
   (发起请求①)          →  ON_RESUME 立即触发
        │                    refreshActiveOrders()
        │                    → loadActiveOrders() (发起请求②)
        │                         │
  请求① 失败 (短暂网络/SSL异常)     请求② 成功
        │                         │
  设置 _errorMessage                设置 _activeOrders
  (Snackbar 显示"加载失败")         (列表正常显示)
```

**两个请求完全独立且同时运行**。请求①（来自 init）失败（如 SSL 首次握手超时、DNS 缓存未就绪），请求②（来自 ON_RESUME）重用已建立的连接成功。

用户看到的现象：Snackbar 一闪而过 "加载取货单列表失败"，但列表实际显示正常数据。

### 现有日志已足够

`loadActiveOrders()` L88 已有 `Log.e("PickListViewModel", "加载取货单列表失败: ${e.message}", e)`，包含完整堆栈和异常消息，无需额外添加日志。

## 二、修改内容

### 文件：`PickListViewModel.kt` — 1 处删除

**L73**：删除 `loadActiveOrders()` 调用

```kotlin
// 修改前
init {
    loadAreas()
    loadActiveOrders()   // ← 去掉这行
    loadCompletedOrders()
}

// 修改后
init {
    loadAreas()
    loadCompletedOrders()
}
```

**为什么安全**：`PickListScreen.kt` L114-L123 的 `DisposableEffect` 注册了 `ON_RESUME` 生命周期观察者。当 ViewModel 创建、Composable 首次组合时，DisposableEffect 同步执行，此时 Activity 已在 RESUMED 状态，`Lifecycle.addObserver` 会**立即回发当前状态事件 ON_RESUME**，触发 `refreshActiveOrders()` → `loadActiveOrders()`。既保证首次加载，也保证页面后续从后台返回时自动刷新。

**影响范围**：仅 PickListViewModel.kt 一行删除，不影响任何其他文件或逻辑。

## 三、回归风险校验

| 风险 | 分析 | 等级 |
|:-----|:-----|:----:|
| 首次进入列表不加载 | `DisposableEffect` + `ON_RESUME` 在首次组合时触发 | ✅ 无风险 |
| 页面从后台返回不刷新 | 同 ON_RESUME 机制处理 | ✅ 无风险 |
| `init` 和 `ON_RESUME` 时序 | `init` 在 ViewModel 构造时执行 → `DisposableEffect` 在 Composable 组合时执行（Activity 已 RESUMED），`ON_RESUME` 确保触发 | ✅ 无风险 |
| 删除 init 中调用不影响其他功能 | `loadAreas()` 和 `loadCompletedOrders()` 保持不变 | ✅ |
| 其他调用 `loadActiveOrders()` 的地方 | `createOrder`/`publishOrder`/`claimOrder`/`confirmDelete` 中仍保留调用 | ✅ |
| 日志诊断 | 已有 `Log.e` 完整堆栈日志 | ✅ 足够 |
