# 商家编码大小写问题修复方案

## 一、问题描述

快麦 SKU 编码为大写字母（如 `B08-24`）。手动输入小写（如 `b08-24`）查询时，快麦 API 大小写不敏感仍能查到数据。但回传备注/供应商时，`erp.item.general.addorupdate` 会使用传入的小写编码覆盖快麦原有的正确大写编码。

## 二、问题链路

```
用户手动输入 "b08-24"（小写）
  ↓
loadSkuInfo("b08-24") → 后端 GET /api/sku/b08-24（system.py L262）
  ↓
get_sku_info("b08-24") → cache.py
  ↓  cache MISS
get_sku_by_outer_id("b08-24") → kuaimai_api.py
  → 快麦返回 { "skuOuterId": "B08-24"，... }
  ↓
get_sku_by_outer_id() mapped 返回：
  { "properties_name": "...", "sys_item_id": ... }
  → 未映射 skuOuterId ❌
  ↓
_cache_data_to_dict("b08-24"， sku_data)：
  { "sku_outer_id": "b08-24"， ... }  ← 用请求参数，不是快麦返回的大写
  ↓
system.py SkuDetailResponse(skuOuterId=sku_outer_id)
  → skuOuterId = "b08-24"  ← 仍用小写
  ↓
Android 显示/存储 skuOuterId = "b08-24"
  ↓
Worker 同步：从 payload 取 skuOuterId = "b08-24" → 传入 updateRemark /
  updateSupplier → 快麦被覆盖为小写
```

## 三、修改方案（4个文件，共5处改动）

### 3.1 后端：kuaimai_api.py — 捕获快麦 API 返回的正确大小写

**文件**: `backend/app/services/kuaimai_api.py` 的 `get_sku_by_outer_id()`
**改动**: 在 mapped 字典中追加 `"sku_outer_id": sku_data.get("skuOuterId", "")`

```python
mapped = {
    "properties_name": sku_data.get("propertiesName", ""),
    "pic_path": sku_data.get("skuPicPath", ""),
    "remark": sku_data.get("skuRemark", ""),
    "sys_item_id": sku_data.get("sysItemId", 0),
    "sys_sku_id": sku_data.get("sysSkuId", 0),
    "item_outer_id": sku_data.get("itemOuterId", ""),
    "supplier_name": "",
    "supplier_code": "",
    "modified": sku_data.get("modified", 0),
    # ↓ 追加
    "sku_outer_id": sku_data.get("skuOuterId", ""),
}
```

### 3.2 后端：cache.py — 缓存写入/读取时优先用 API 返回的大小写

**文件**: `backend/app/services/cache.py`

**改动 A** — `get_sku_info()` L52-54（缓存命中但未变更时，优先用 API 返回的大小写）：

```python
# 修改前：
if cached and api_modified == cached_modified:
    logger.debug(f"SKU数据未变更(modified={api_modified})，返回缓存: {sku_outer_id}")
    return _cache_row_to_dict(cached)

# 修改后：
if cached and api_modified == cached_modified:
    logger.debug(f"SKU数据未变更(modified={api_modified})，返回缓存: {sku_outer_id}")
    cached_dict = _cache_row_to_dict(cached)
    # 优先用快麦 API 返回的正确 sku_outer_id 大小写
    api_sku_outer_id = sku_data.get("sku_outer_id")
    if api_sku_outer_id:
        cached_dict["sku_outer_id"] = api_sku_outer_id
    return cached_dict
```

**改动 B** — `_api_data_to_dict()` L131（API 响应转字典时，优先用 data 中的 sku_outer_id）：

```python
# 修改前：
"sku_outer_id": sku_outer_id，
# 修改后：
"sku_outer_id": data.get("sku_outer_id"， sku_outer_id)，
```

> `_cache_row_to_dict()` L116 保持原样：缓存行的 `sku_outer_id` 可能是请求参数大小写，但 L52-54 的改动 A 已在缓存命中时用 API 返回值覆盖。API 异常降级场景（L39、L45）可接受返回缓存大小写，因为 API 不可用时同步操作也大概率会失败。

### 3.3 后端：system.py — 返回真实 skuOuterId

**文件**: `backend/app/routers/system.py`
**改动**: `get_sku_detail()` L293：

```python
# 修改前：
skuOuterId=sku_outer_id，
# 修改后：
skuOuterId=sku_info.get("sku_outer_id"， sku_outer_id)，
```

### 3.4 Android：OrderSyncWorker.kt — 使用 API 返回的正确 skuOuterId

**文件**: `app/src/main/java/com/kuaimai/pda/data/OrderSyncWorker.kt`

**改动 A** — `SkuSyncData` 增加 `skuOuterId` 字段：

```kotlin
private data class SkuSyncData(
    val title: String，
    val itemOuterId: String，
    val propertiesName: String，
    val skuOuterId: String = ""  // 追加
)
```

**改动 B** — `fetchLatestSkuDataViaBackend()` 填充 `skuOuterId`：

```kotlin
// 修改后：
SkuSyncData(
    title = title,
    itemOuterId = itemOuterId,
    propertiesName = detail.propertiesName,
    skuOuterId = detail.skuOuterId  // 追加
)
```

**改动 C** — `fetchLatestSkuData()`（直连快麦路径）也填充：

```kotlin
// 两个 return 都追加：
skuOuterId = sku?.skuOuterId ?: ""
```

**改动 D** — `syncRemarkUpdate()` 使用 API 返回的大小写：

```kotlin
// 在构建 request 之前：
val effectiveSkuOuterId = skuOuterId  // 来自 payload
val correctSkuOuterId = if (skuData.skuOuterId.isNotBlank()) skuData.skuOuterId else effectiveSkuOuterId
// 用于 SkuUpdateDto：
skus = listOf(SkuUpdateDto(skuId = skuId, skuOuterId = correctSkuOuterId， ...))
```

**改动 E** — `syncSupplierUpdate()` 同理：

```kotlin
val correctSkuOuterId = if (skuData.skuOuterId.isNotBlank()) skuData.skuOuterId else skuOuterId
skus = listOf(SkuUpdateDto(skuId = skuId, skuOuterId = correctSkuOuterId， ...))
```

## 四、回归风险评估

| 风险点 | 评估 |
|:-------|:-----|
| `data.get("sku_outer_id")` 在首次部署后的缓存命中场景下为空 | 回退到 `sku_outer_id`（请求参数），不损坏数据 |
| `detail.skuOuterId` 为空白 | Worker 回退到 payload 的 skuOuterId |
| `_cache_row_to_dict` 中 `sku_outer_id` 未改 | 已在 L52-54 缓存命中路径用 API 返回值覆盖；仅 API 异常降级时返回缓存大小写，此场景下同步大概率失败，可接受 |
| Worker 直连快麦路径 `fetchLatestSkuData` 中 `sku?.skuOuterId` | `SkuItemInfo` 有 `skuOuterId` 字段（`KuaimaiQueryDto.kt` L30），`?.` 安全调用 + `?: ""` fallback，不会 NPE |
| Worker 后端路径 `fetchLatestSkuDataViaBackend` 中 `detail.skuOuterId` | `SkuDetailResponse` 有 `skuOuterId` 字段（`SkuDetailResponse.kt` L8），默认值为 `""`，不会 NPE |
| 缓存表 `sku_outer_id` 主键仍为请求参数 | 不影响查询匹配，不影响返回给调用方的值 |

## 五、不修改的文件

| 文件 | 原因 |
|:----|:-----|
| `cache.py` save_sku_to_cache | INSERT 的 sku_outer_id 主键用请求参数（查询必须匹配），不影响返回 |
| 数据库 schema | 无需加列，完整性足够 |
| `ProductViewModel.kt` | UI 层显示用户输入不影响更新 |
| `PickOrderRepository.kt` | payload 存储原始输入是合理设计 |
| 管理后台 | 无变化 |
