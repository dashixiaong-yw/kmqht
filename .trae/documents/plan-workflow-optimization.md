# 计划：优化开发修改流程 - 支持批量任务模式

## 问题分析

当前流程是**单次修改的线性8步流程**：

```
查阅知识图谱 → 修改代码 → 验证 → 构建APK → 更新版本号 → 更新知识图谱 → 同步docker-deploy → Git提交
```

**核心问题**：当有多个任务时，当前流程没有"批量模式"的概念。AI可能：

1. **逐任务走完整流程**：每个任务都执行8步，导致版本号频繁递增，产生大量小提交
2. **中间提交后封版**：按封版规则"禁止同版本二次修改"，Git提交后无法继续修改，后续任务被阻断
3. **跳过后续步骤**：因为封版限制，AI可能直接跳过剩余任务

虽然封版规则中有"批量修复原则"，但描述模糊：
- "同一任务"定义不清
- 流程步骤表没有体现"批量修改"的循环结构
- Step 2"修改代码"是单次动作，没有说明可以重复
- 缺少AI判断"何时进入收尾阶段"的明确条件

## 优化方案

将流程拆分为**两个阶段**：开发阶段（可循环）和收尾阶段（一次性不可逆），并增加AI执行的硬性判断规则。

### 新流程结构

```
┌──────────────────── 开发阶段（可循环） ────────────────────┐
│ Step 1: 查阅知识图谱（首次必须，后续按需）                  │
│ Step 2: 修改代码（支持批量，全部完成后进入收尾）             │
│ Step 3: 验证代码（lint）                                    │
│ Step 4: 构建APK                                            │
│ ─── 若验证/构建失败，回到Step 2修复 ───                     │
└────────────────────────────────────────────────────────────┘
                          │
                全部任务完成 + 验证构建通过
                          ▼
┌──────────────────── 收尾阶段（一次性，不可逆） ───────────┐
│ Step 5: 更新版本号（3处一致+验证）                          │
│ Step 6: 更新知识图谱                                       │
│ Step 7: 同步到docker-deploy                                │
│ Step 8: Git 提交推送                                       │
└────────────────────────────────────────────────────────────┘
```

### AI执行硬性约束（写入规范文档）

**约束1：收尾阶段准入条件**
进入Step 5前，必须同时满足以下3个条件：
- ✅ 所有任务代码修改已完成（逐项确认，无遗漏）
- ✅ `./gradlew lint` 通过
- ✅ `./gradlew assembleDebug` 构建成功

**约束2：收尾阶段禁止回退**
进入Step 5后，禁止再修改任何代码文件。若发现遗漏，必须完成当前收尾（Step 5-8），再以新版本号重新走完整流程。

**约束3：批量任务必须合并提交**
多个任务（无论P0/P1/P2）必须在同一版本中合并提交，禁止每个任务单独提交。版本号只递增一次。

**约束4：CHANGELOG合并记录**
批量任务的CHANGELOG按"新增/修改/修复"分类合并记录，不按任务逐条拆分版本。

### 具体修改文件

**文件**：`.trae/rules/README.md`

**修改1**：第二节"开发修改流程"——替换流程图和步骤表

箭头图改为：
```
[查阅知识图谱] → [修改代码(可循环)] → [验证+构建(失败回退)] → 全部完成 → [更新版本号 → 更新知识图谱 → 同步docker-deploy → Git提交推送]
```

步骤表改为两阶段（增加"阶段"列）：

| Step | 阶段 | 内容 | 说明 |
|:----:|:----:|------|------|
| 1 | 开发 | **查阅知识图谱** | 首次修改前必须查阅；批量任务中后续任务按需查阅 |
| 2 | 开发 | 修改代码 | 在 `app/` 目录修改，**支持批量完成多个任务后再进入收尾** |
| 3 | 开发 | 验证代码 | `./gradlew lint` 必须通过；**失败则回到Step 2修复** |
| 4 | 开发 | 构建APK | `./gradlew assembleDebug` 构建成功；**失败则回到Step 2修复** |
| 5 | 收尾 | 更新版本号 | **⚠️ 进入收尾后禁止再修改代码**；更新3处版本号并验证一致 |
| 6 | 收尾 | **更新知识图谱** | 将本次所有变更的设计决策同步到知识图谱 |
| 7 | 收尾 | 同步到docker-deploy | 运行同步脚本 |
| 8 | 收尾 | Git 提交推送 | `git add .` → `git commit` → `git push` |

**修改2**：封版规则——重写为更明确的规则

| 规则 | 说明 |
|------|------|
| **开发阶段可循环** | Step 1-4可重复执行，直到所有任务完成且验证构建通过 |
| **收尾阶段不可逆** | 进入Step 5后禁止再修改任何代码，必须完成Step 5-8 |
| **批量任务合并提交** | 多个任务必须在同一版本中合并提交，版本号只递增一次 |
| 禁止同版本二次修改 | Git 提交后，该版本代码封版 |
| 新变更必须新版本 | 提交后发现需要修改，必须递增版本号，重新走完整流程 |
| 禁止重复版本号 | CHANGELOG.md 中同一版本号只能出现一次 |

**修改3**：验证清单——增加批量任务检查项和阶段标识

```
开发阶段：
- [ ] 已查阅知识图谱确认设计决策
- [ ] 所有任务代码修改已完成（批量任务时逐项确认无遗漏）
- [ ] 代码修改在 `app/` 目录完成
- [ ] 代码验证通过（lint）
- [ ] APK构建成功（app-debug.apk 产出）

收尾阶段：
- [ ] 3处版本号一致（build.gradle.kts + CHANGELOG.md + gradle.properties）
- [ ] 已更新知识图谱（本次所有变更的设计决策已同步）
- [ ] 已同步到docker-deploy
- [ ] Git commit 消息符合格式 `v版本号: 变更描述`
- [ ] CHANGELOG.md 已更新
```

## 验证步骤

2. 确认步骤编号与知识图谱强制步骤引用（Step1和Step6）一致
3. 模拟批量任务场景（如11个任务），确认流程不会在中间封版
