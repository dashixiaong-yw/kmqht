## 二、根因分析 — 3 个核心问题

### 问题 1：Plan Mode 系统指令与项目规则分离（最根本原因）

**系统 Prompt 中的 Plan Mode 流程**：
```
Phase 1: Explore (搜索代码)
Phase 2: Intent chat (问用户意图)
Phase 3: Implementation chat (问实现细节)
Phase 4: Generate Plan (写 .trae/documents/*.md)
Phase 5: NotifyUser → 用户批准 → 执行
```

**项目规则中的开发流程**：
```
Step 1: 查阅知识图谱
Step 2: 修改代码
Step 3: lint + 预构建
Step 4: 更新版本号
Step 5: 最终构建APK
Step 6: 更新知识图谱
Step 7: 同步docker-deploy
Step 8: Git提交推送
```

**两者完全不打通**。系统 Prompt 的 Phase 5"用户批准→执行"对项目规则 Step 1-8 一无所知。AI 在 Phase 5 进入执行模式后，没有机制强制它遵循项目规则的 8 步流程。这导致：

```
正常应该:
  Plan Mode（出方案）→ 用户批准 → 进入项目规则 8 步流程

实际发生了:
  Plan Mode（出方案）→ 用户批准 → AI 只改了代码就直接 git commit
    → 跳过了：知识图谱查阅、lint、版本号、构建、同步
```

### 问题 2：规则缺少"Plan Mode 结束后如何接入执行阶段"的衔接说明

`.trae/rules/README.md` 中完全没有提及 Plan Mode。当系统 Prompt 的 Plan Mode 结束时，AI 进入了"执行模式"，但项目规则没有告诉 AI：

- "用户批准后，你应进入项目规则的 Step 1（查阅知识图谱）"
- "用户批准的方案已经是全部需求，收尾阶段禁止新增代码"

### 问题 3：逐个需求的迭代模式 vs 批量合并提交流程冲突

规则要求"批量任务必须合并提交"、"收尾阶段进入后禁止回退"。但用户习惯是**逐个提需求**：

```
用户: 修复 A
  → AI: 修复 A，构建，提交 v1.97 ✅（正确）
用户: 删掉多余文件
  → AI: 删文件，直接 git commit ❌（应走 v1.98 完整流程）
用户: 再改个 B
  → AI: 改 B，直接 git commit ❌（应合并到 v1.98）
用户: 再改个 C
  → AI: 改 C，直接 git commit ❌（同上）
```

规则没有被违反的"故意"，而是**迭代模式下自然而然的走捷径**。规则没有明确说明："即使只改 1 行代码、用户只提了 1 个小需求，也**必须**走完整 8 步流程"。

## 三、规则改进建议

### 改进 1：在 .trae/rules/README.md 中新增"Plan Mode 衔接说明"

在"二、开发修改流程"之前新增：

```markdown
## 计划模式（Plan Mode）衔接规范

当 AI 处于 Plan Mode 生成方案时：

1. **方案审核标准**：方案必须包含完整 8 步流程的规划（知识图谱 → 代码 → lint → 版本号 → 构建 → 知识图谱 → 同步 → Git）
2. **用户批准 = 全部需求锁定**：批准后此次迭代的所有需求已确定，禁止在收尾阶段新增任何代码变更
3. **方案批准后的执行路径**：
   - 用户批准后，自动进入下方「开发修改流程」的 Step 1
   - 严格遵守 8 步流程，禁止跳过任何一步
   - 所有任务完成后进入收尾，禁止中途退出收尾回修改代码
```

### 改进 2：强化 Step 4-8 的强制检查

在 Step 表 footer 新增：

```markdown
**强制检查（收尾阶段前必须逐条确认）**：

- [ ] 所有代码修改已完成（对照批准的方案逐项检查）
- [ ] 知识图谱已查阅（Step 1 已执行）
- [ ] lint + 预构建通过（Step 3 已执行）
- [ ] 版本号已递增（Step 4）
- [ ] 最终 APK 文件名 = 新版本号（Step 5）
- [ ] 知识图谱已更新（Step 6）
- [ ] docker-deploy 已同步（Step 7）
- [ ] Git commit 消息格式 = `v版本号: 变更描述`（Step 8）
```

### 改进 3：明确"小修改也要走完整流程"

在 Step 4 说明中追加：

```markdown
> **重要**：即使只修改 1 行代码，也必须递增版本号走完整流程。禁止"无版本号的热修复提交"。
> 禁止 commit 消息不使用 `v版本号:` 前缀的任何提交。
```
