# 消除双重构建：将版本号更新移至开发阶段

## 一、当前问题

修正后的规则仍然执行**两次 assembleRelease**：

```
开发阶段：
  Step 3: lint + assembleRelease (预检)   ← 第1次构建 (~3min)
收尾阶段：
  Step 4: 更新版本号
  Step 5: assembleRelease (最终APK)       ← 第2次构建 (~3min)
```

每次发版浪费 **~6 分钟构建时间**，而两次构建的产物完全相同（仅版本号变了）。

## 二、优化方案

**将「更新版本号」从收尾阶段移到开发阶段**，收尾阶段仅构建一次：

```
开发阶段（可循环 Step 1-4）：
  Step 1: 查阅知识图谱
  Step 2: 修改代码       ←──────────┐
  Step 3: ./gradlew lint             │ 失败循环
  Step 4: 更新版本号 (4处手动)       ┘
     ↓
收尾阶段（不可逆 Step 5-8）：
  Step 5: ./gradlew assembleRelease（唯一一次构建，APK含新版本号）
  Step 6: 更新知识图谱
  Step 7: 同步docker-deploy
  Step 8: Git提交推送
```

### 关键收益

| 维度 | 当前（双重构建） | 优化后（单次构建） |
|:-----|:---------------|:---------------|
| assembleRelease 次数 | 2次 | 1次 |
| 每次发版构建耗时 | ~6 min | ~3 min |
| 构建失败是否需要递增版本 | 是（Step 4已锁定） | 否（仍在开发阶段可循环） |

### 边界情况

- **构建失败**：仍在开发阶段（Step 4 在开发阶段），可直接修改代码后重新 lint → 版本号不变 → rebuild，无需递增版本号
- **构建成功**：进入收尾（Step 5），产物 APK 已含正确版本号，Step 6-8 照常执行
- **收尾后发现问题**：Git 已提交，需递增版本号重新走完整流程（封版规则不变）

## 三、规则文本修改

### 修改1：流程图

```
[查阅知识图谱] → [修改代码(可循环)] → [lint验证] → [更新版本号] → 全部完成 → [构建APK → 更新知识图谱 → 同步docker-deploy → Git提交推送]
```

### 修改2：Step 表

| Step | 阶段 | 内容 | 说明 |
|:----:|:----:|------|------|
| 1 | 开发 | **查阅知识图谱** | 首次修改前必须查阅；批量任务中后续任务按需查阅 |
| 2 | 开发 | 修改代码 | 在 `app/` 目录修改，**支持批量完成多个任务后再进入收尾** |
| 3 | 开发 | 验证代码 | `./gradlew lint` 必须通过；**失败则回到Step 2修复** |
| 4 | 开发 | 更新版本号（含Docker BUILD_VERSION） | **先读取6处当前版本号取最大值，再+1递增**，更新6处并验证一致。**升级后仍可回到Step 2修复代码** |
| 5 | 收尾 | 构建APK | `./gradlew assembleRelease`（签名+混淆）构建成功，**此APK为最终发布版本**（文件名含新版本号）；**失败则回到Step 2修复** |
| 6 | 收尾 | **更新知识图谱** | 将本次所有变更的设计决策同步到知识图谱 |
| 7 | 收尾 | 同步到docker-deploy | 运行 `.\scripts\sync-to-docker-deploy.ps1 -Force` 同步后端部署文件 |
| 8 | 收尾 | Git 提交推送 | `git add .` → `git commit -m "v版本号: 变更描述"` → `git push` |

### 修改3：准入条件

```
**收尾阶段准入条件（进入Step 5前必须同时满足）**：

- ✅ 所有任务代码修改已完成（逐项确认，无遗漏）
- ✅ ./gradlew lint 通过
- ✅ 版本号已更新（6处一致）
```

### 修改4：封版规则

| 规则 | 说明 |
|------|------|
| **开发阶段可循环** | Step 1-4可重复执行，直到所有任务完成且验证通过 |
| **收尾阶段不可逆** | 进入Step 5后禁止再修改任何代码，必须完成Step 5-8 |

### 修改5：禁止回退描述

```
**收尾阶段禁止回退**：进入Step 5后禁止再修改任何代码文件。若发现遗漏，必须完成当前收尾（Step 5-8），再以新版本号重新走完整流程。
```
